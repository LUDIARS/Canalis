Canalis — Ci

LUDIARS 共有 データ取込パイプライン基盤（ラテン語で「管・水路」）

外部データを ① crawl/scrape → ② clean/categorize → ③ save/import の 3 ステージで取り込む共有ライブラリ。 各ステージは型（契約）で疎結合に接続され、サービスはマニフェストで流れを宣言する。 Tirocinium の Notion クローラと Discutere の外部コメント収集を共通基盤へ載せ替えるために起こされた。

サービスの役割

Canalis は「集める → 整える → 保存する」を 1 本のパイプラインとして抽象化し、 その骨格（契約・汎用 adapter/writer・実行 runner）だけを共有する。 サービス固有の整形ロジックは持たず、各サービスが「②Transform 1 本 + マニフェスト 1 枚」を足すだけで 新しい取込パイプラインを宣言できる。

① 取得を共通化

Notion / YouTube / Reddit / Website / Discord などの取得元 adapter を共有。被れば流用、無ければ 1 つ足す。

② 整形は各サービス

サービス内部仕様への整形・分類は各リポの責務。共有 lib には 1 サービス専用ロジックを置かない。

③ 保存を共通化

raw(JSONL) / Postgres / Kuzu / FT の writer を共有。②が作った最終形を verbatim に書くだけ。

再処理 (replay)

①→② が常に RawRecord[] なので、raw store から ②→③ を再クロールせず焼き直せる。

3 ステージ

① CRAWL / SCRAPE

取得（汎用）

Source 実装が取得元から RawRecord[] を得る。adapter を足せば取得元が増える。

→
② CLEAN / CATEGORIZE

整形（サービス固有）

Transform が sink ネイティブの最終形へ整形・分類・付与する。各サービスリポに置く。

→
③ SAVE / IMPORT

保存（汎用）

Sink 実装が保存先へ verbatim 書込。table-map も射影も持たない。

① Crawl/Scrape ──RawRecord[]──┬──▶ raw sink (verbatim 保存)        [任意]
   汎用 (adapter)              │
                              └──▶ ② Clean/Categorize ──sink-native──▶ ③ Save/Import
                                    サービス固有 (plugin)              汎用 (writer)

関係の全体像と各機能の依存は グラフビュー で、 公開関数・型・パラメータは API リファレンス で確認できる。

設計原則（鉄則）

1. LLM をできる限り排除する（最重要）。
共有 lib（@ludiars/canalis）は LLM への依存を一切持たない（SDK も API キーも要求しない）。 整形・分類は決定論的手段を第一とする（セレクタ / 正規表現 / パーサ / 辞書 / ルール）。 LLM が避けられない処理は、サービス側 ②の中で明示的な opt-in アダプタとして実装する。 opt-in ヘルパは別サブパス @ludiars/canalis/llm に隔離し、root の公開面（core/crawl/save）には混ぜない。
2. 整形の知識は ②に集約し、③は verbatim 書込。
②が「保存先のネイティブ最終形」まで作り切る。③は渡された形をそのまま書くだけで、 table-map も射影もサービス知識も持たない。→ ③writer は sink エンジン 1 種につき 1 個で全サービス共有。
3. 共有 lib は 1 サービス専用ロジックを持たない。
① adapter（汎用）と ③ writer（汎用）と契約・runner だけを共有。②Transform は各サービスリポに置く。
4. ③writer に DB ドライバを直接依存させない。
SqlExecutor / CypherExecutor を注入する（共有 lib は postgres/kuzu に直接依存しない）。 FT も FtRunner を注入し、学習ランタイム（Python/PEFT 等）に直接依存しない。

主な特徴

特徴内容
契約による疎結合Source / Transform / Sink の 3 interface と、RawRecord / SinkBatch エンベロープのみで接続。
マニフェスト駆動サービスはコードでなく宣言的な manifest（YAML/JSON）で流れを定義。runPipeline(manifest, deps) が解釈実行。
依存注入 (DI)source / transform / sink / DB driver / LLM / FT runner はすべて呼び出し側が注入。core はグローバル registry を持たない。
依存ゼロのクローラ各 adapter は raw fetch（+ 任意で Playwright）で実装。SDK 依存なしで移植・流用が容易。
replay（再処理）raw store の RawRecord[] を起点に ②→③ を焼き直せる。rate-limit / コスト / 元データ消失に強い。
安価推論カスケード@ludiars/canalis/llm に Tier0 決定論 → Tier1/2 注入モデルの段階分類。LLM は最後の手段。
学習的インポート (FT)収集データをローカル LLM の fine-tune 教師として materialize（分類器 / 生成 LLM の両対応）。学習は core 外 runner。
secret 暗号化Notion token は config.json に salt 付き AES-256-GCM で暗号化保存（非シークレットは平文）。

レビュー結果 — spec ↔ コード対応

設計ドキュメント（DESIGN.md / README.md / CLAUDE.md）と実コードを突き合わせた結果。 設計原則はコードに忠実に守られている一方、機能カタログの記述がコードに追いついていない箇所がある。 本サイトは現行コードの実態を反映してまとめている。

項目状態所見
原則: LLM 隔離（root に混ぜない）一致src/index.ts は llm/ を export しない。LLM は /llm サブパスのみ。
原則: ②を共有 lib に置かない一致core/crawl/save/runner のみ。Transform は契約 (interface) だけ。
原則: ③ verbatim + driver 注入一致Postgres/Kuzu は SqlExecutor/CypherExecutor 注入。直接依存なし。
契約（RawRecord / エンベロープ / manifest）一致DESIGN §2–3 の型がコードと一致。
① crawl adapter 群ドキュメント遅延DESIGN §5/README は「notion のみ」。実際は 7 adapter（notion, notion-public, youtube, reddit, website, discord, discord-guild）。discord / notion-public は DESIGN.md に記載なし。
LLM カスケード（cascade/lexicon/scorer）ドキュメント遅延DESIGN §0.1 は localOpenAiExecutor のみ記載。実際は Cascade / lexiconScorer / executorScorer も実装済（未記載）。
manifest ローダ（YAML/JSON）記述が古いDESIGN §6 で「未了」だが loadManifest は実装＋テスト済。
DB 結合テスト一部Kuzu 結合テストは通電（実行・成功）。Postgres 結合テストは live DB が無いため skip。
FT runner 実体一部TS 側 FtSink は実装＋テスト済。Python runner は causal-lm(Gemma) のみ、classification は別 runner（未）。
推奨フォローアップ: DESIGN.md §5 の adapter 一覧と §0.1 の LLM 節を現状（7 adapter ＋ cascade サブシステム）に追従し、 §6 の「未了」から manifest ローダ／Kuzu 結合テストを完了側へ移す。

使い方（現 scaffold v0.1）

npm install
npm run build
npm test          # fake で ①→② 契約を検証

# Notion token を暗号化して config.json に保存
npm run cli -- config set notion.token
npm run cli -- config show

# ① crawl + raw 保存だけの standalone
npm run cli -- notion-crawl <databaseId> --out data/raw

プログラムからフルパイプライン:

import { runPipeline, NotionSource, PostgresSink, JsonlRawSink } from '@ludiars/canalis';

const report = await runPipeline(manifest, {
  sources:    { notion: new NotionSource() },
  transforms: { 'tr:company-normalize': myTransform },  // ② はサービス側
  sinks: {
    jsonl:    new JsonlRawSink({ dir: 'data/raw/tr' }),
    postgres: new PostgresSink(mySqlExecutor),
  },
});
